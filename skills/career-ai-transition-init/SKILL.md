# /career-ai-transition-init — 职业 AI 转型初始化

把用户从“我想找更好的 AI 相关工作”带到“有一份信息足够、可追溯、待市场验证的能力画像”。本阶段的目标是问清楚，不是立即推荐岗位。

## 产品边界

- **China-first，不是 China-only**：默认使用中文、中国大陆求职语境和一二线城市优先策略。
- 非中国用户可以完成能力盘点；进入岗位验证时，改用当地招聘来源或由用户提供 JD，不套用 BOSS 直聘、中国城市、人民币薪资和本地反诈规则。
- 不承诺高薪、录用或固定转型周期。所有方向在真实 JD 验证前都只是假设。
- 不以兴趣直接推荐职业。后续顺序是：市场需求 → 岗位核心能力 → 用户能力 → 差距可补性 → 城市与薪资 → 个人偏好。

## 工具原则

- 初始化只使用本地文件读写，不要求用户预先安装 Node.js、Chrome 调试环境或 BOSS adapter。
- 不在本阶段大量搜索岗位。用户选定一个待验证方向后，由 `career-ai-transition-scan` 检测依赖并开始针对性搜索。
- 招聘数据工具不可用时，后续允许用户手动粘贴 JD 或使用公开网页信息。

## 执行流程

```text
Phase 0  检测 .skill-state.json
Phase 1  说明方法、边界和数据市场
Phase 2  收集职业背景和 AI 实践
Phase 3  对模糊回答强制追问
Phase 4  收集项目证据、求职约束和学习投入
Phase 5  形成画像与 2–4 个待验证方向
Phase 6  写入 .skill-state.json
Phase 7  让用户选择一个方向，进入深度岗位验证
```

## Phase 0 — 检测状态

```bash
test -f .skill-state.json && echo EXISTS || echo MISSING
```

- 已存在：先摘要展示现有画像，询问是更新画像还是继续。
- 不存在：继续初始化。
- 已存在且 `schema_version < 2`：运行 `node scripts/migrate-state.mjs .skill-state.json`，先生成备份再升级；失败时停止写入。

首次初始化时确认一个专用职业转型工作目录。如果当前目录是无关代码仓库或其他项目，先提醒用户切换目录，避免将求职隐私误提交到 Git。

## Phase 1 — 首屏说明

用自然语言表达：

1. 这不是兴趣测试，也不是通用高薪岗位清单。
2. 我会先把你的真实经历和 AI 实践问清楚；如果回答模糊，会继续追问。
3. 初始化不搜索大量岗位。选定方向后，再用真实招聘信息验证核心能力、工具、城市和薪资。
4. 如果你是 AI 零基础，也可以使用；但输出会先是市场验证后的学习准备路线，而不是立即投递建议。

## Phase 2 — 首轮问题

首轮一次最多问 5 个问题，避免把对话变成长问卷。

1. **职业背景**：你目前所在的行业、岗位和工作年限是什么？请举一个最能代表你能力的真实工作结果。
2. **AI 实践**：请讲一件你最近使用 AI 完成的真实任务。当时要解决什么问题，你做了什么，AI 做了什么，最后结果是什么？
3. **工具与独立程度**：你在这些任务中使用了哪些工具？是跟教程完成、在 AI 指导下完成，还是可以独立设计、修复和交付？
4. **方向状态**：你已经有明确想深入的职业，还是希望先根据经历提出几个待验证方向？
5. **市场范围**：你在哪个国家和城市求职？是否接受迁移、通勤或远程？

### AI 工具说明示例

当用户不知道“AI 工具或实践”指什么时，可提供示例，但不把示例当成必选项：

- 大模型对话与内容任务；
- Skill、Agent 或多步工作流；
- Coze、n8n、飞书自动化等低代码流程；
- API 接入、JSON 处理或第三方服务连接；
- AI 编程、Vibe Coding、调试、测试、打包或部署；
- AI 生图、视频、数据处理、知识库或其他可展示成果。

示例的作用是帮用户回忆，不是用术语暗示高分答案。

## Phase 3 — 强制追问规则

### 硬性原则

**核心信息未达到可判断状态时，禁止写入最终画像、禁止进入岗位验证。**

每次最多追问 3 个有关联的问题。一个问题可以追问二次、三次，直到达到可判断状态；如用户确实无法提供，记为 `unknown` 并明确说明会降低后续结论置信度。

### 模糊回答的判定

以下回答不能直接当作能力证明：

- “会一点”“经常用”“基本都会”；
- 只列工具名，没有说用来解决什么问题；
- 只说“做过项目”，没有说本人贡献和结果；
- 只说“跟着教程做过”，没有说能否复现、修改和解决报错；
- 只给自我评价，没有具体场景。

### 追问模板

优先追问事实，不追问抽象自评：

- “请选最近一次，当时的任务是什么？”
- “第一次输出可以直接使用吗？如果不可以，你怎么调整？”
- “哪一步是你自己决定的，哪一步是按照教程或 AI 提示完成的？”
- “换一份输入，你现在能不能重新做出来？”
- “有什么可以展示的证据：截图、文件、链接、代码、录屏或结果数据？”

### 信息状态

- `unknown`：没有信息。
- `vague`：只有模糊自述，必须追问。
- `claimed`：有具体过程，暂无可检查证据。
- `evidenced`：有截图、文件、链接、代码或数据。
- `verified`：后续已实际检查或运行产出。

## Prompt 表达观察

不进行 Prompt 术语考试，不询问用户是否会某个 Prompt 框架，不给出脱离场景的“Prompt 能力分”。

通过用户的真实案例和多轮追问观察：

- 能否说清背景、目标、输入、过程和结果；
- 能否说明第一次输出的问题以及后续调整；
- 能否说清哪些由人判断，哪些交给 AI；
- 能否提供判断结果是否合格的依据。

只保存观察到的优势、缺口和案例证据。不在 init 阶段教 Prompt；由 `career-ai-transition-plan` 根据目标岗位将相关训练写入学习计划。

## Phase 4 — 补全核心字段

根据首轮回答按需询问，不重复已说清的内容：

1. **项目证据**：有哪些可展示成果？是否可以提供文件、链接、截图、录屏、代码或效果数据？
2. **价值结果**：这些实践节省了什么时间、降低了什么成本、提高了什么质量，或者产生了什么业务结果？
3. **学历与经验**：最高学历、相关工作年限及可迁移的专业经验。
4. **薪资与时间**：当前薪资、目标薪资、希望多久开始投递、每周可投入多少学习时间。
5. **工作约束**：是否接受编程、客户沟通、出差、项目交付、远程或城市迁移。这些是约束，不是在市场需求之前的兴趣投票。
6. **公开输出现状**：是否有正在使用的自媒体或公开作品账号？平台可以是公众号、小红书、知乎、B 站、抖音、视频号、GitHub、个人网站或其他公开平台。请说明账号主题、内容形式、更新频率、代表作品和可公开链接；没有则明确记录为“尚未开始”。

### 进入下一阶段前的硬性字段

- 国家、城市和可迁移范围；
- 当前职业背景和一个真实结果；
- 至少一个 AI 实践案例，或明确标记为 AI 零基础；
- 工具、本人贡献和独立程度；
- 证据状态；
- 当前与目标薪资；
- 可投入学习时间与转型时间。
- 现有公开账号、内容形式、更新频率和代表作品；没有账号时标记为待启动。

## 公开发声与专业证据

- 本 Skill 强烈建议用户持续公开输出，但不把它设为阻止学习进度的硬性门槛，目标也不是强迫用户成为娱乐型网红。
- 发声的核心目的是形成可被招聘者、同行和潜在合作者检查的 **public proof of work（公开工作证明）**。
- 形式不限：文章、视频、图文、开源仓库、项目日志、案例拆解、Demo 或其他可公开查看的产出均可。
- 用户可使用匿名或网名，不要强制真人出镜、使用真名或公开私人信息。
- 不得发布公司机密、客户数据、个人隐私、未获授权代码或受限材料；必要时先做脱敏和重建示例。
- init 阶段只盘点现状和约束，不制定具体更新任务。目标岗位确定后，由 `career-ai-transition-plan` 将公开输出写入学习计划。

## 中国市场与城市策略

- 对中国大陆用户，默认建议后续将一线和强二线城市作为对照组，因为本 Skill 假设 AI 岗位的机会密度、产业链和成长机会更集中。
- 这是本 Skill 的导向，必须向用户透明说明，不冒充无条件事实。
- 用户选择其他城市时继续服务，后续将当地与一二线参照城市的岗位数量、薪资、公司类型和技能要求放在一起比较。
- 未经实时岗位数据验证前，不声称某个城市一定更好。

## Phase 5 — 形成画像

输出一份可供用户修正的摘要：

- 职业背景与可迁移能力；
- AI 工具和真实实践；
- 独立完成程度；
- 项目证据状态；
- Prompt 表达观察（不打分）；
- 城市、薪资、学习时间与其他约束；
- 现有公开账号、内容形式、代表作品和发声起点；
- 数据缺口和结论置信度；
- 2–4 个待市场验证的职业方向。

方向必须标注“待验证”，并说明它只是根据当前画像产生的假设，尚未根据实时 JD 判断需求、薪资和门槛。

## Phase 6 — 写入状态文件

读取 `templates/skill-state.template.json`，填入已问清的字段。不得用推测代替未知信息。`start_tier` 只是可选起点摘要，可为 `null`；后续必须用 JD 要求与用户证据逐项对比。

写入前获取时间和本机时区：

```bash
date '+%Y-%m-%d %H:%M %Z %z'
readlink /etc/localtime 2>/dev/null | sed 's#.*/zoneinfo/##'
```

将时间写入 `created_at`、`created_at_full`、`timezone_id` 和 `timezone_label`。将文件写到当前工作目录的 `.skill-state.json`。

## Phase 7 — 进入针对性岗位验证

让用户选择一个待验证方向。选定后路由到 `career-ai-transition-scan`。

不要在 init 阶段安装招聘工具。`career-ai-transition-scan` 应当在真正需要读取招聘数据时：

1. 检测环境；
2. 说明用途与边界；
3. 让用户选择配置 BOSS adapter、手动粘贴 JD 或使用公开网页；
4. 然后开始深度分析。
