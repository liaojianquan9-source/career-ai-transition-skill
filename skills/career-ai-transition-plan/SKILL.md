# /career-ai-transition-plan — 个性化学习策略（差距 → 路径 → 上岸）

把一个目标岗位变成"以你的起点 + 每周时间，几个月内能学到能投简历"的可执行学习计划。

## 前置检查
- 若 `schema_version < 2`，先运行 `node scripts/migrate-state.mjs .skill-state.json`备份并升级。
- 读 `.skill-state.json` 找该岗位。若不在 `candidate_roles` 中，先回 `career-ai-transition-scan` 完成真实 JD 验证。若 `transition_verdict` 为“高成本”或“当前不建议”，先解释证据和代价，由用户明确确认后才可继续。
- 若该岗位牵涉付费培训/内推，先确认它过了 `references/anti-scam-rubric.md`；命中红线先劝退。
- 读 `references/learnability-rubric.md`（差距维度对齐）和 `references/role-tiers.md`（只作可选起点摘要，不取代逐项证据对比）。

## 学习策略必含 7 块

1. **差距分析（最重要）**：把目标岗位 JD 要求 − 用户已有能力 = 要补的清单，逐条列，**标出哪些可迁移（已有，略学即可）、哪些要从头学**。差距清单决定整个计划，别跳过。

2. **分阶段学习路径**：拆成 2–4 个阶段，每阶段有：
   - 目标（学完能做什么，可验证）
   - 具体内容 + 推荐资源类型（**优先免费/低成本 + 边学边做**；要推具体课程/资源时用 WebSearch 查当下真实可得的，加年份，别凭记忆给可能失效的链接）
   - **AI 加速点**：这一阶段怎么用 AI 当私教 / 用 AI 编程 / 用 AI 出作品，把学习曲线压短（这是 AI 时代学习的题眼，别让"用 AI"停在口号）
   - 时间盒（以用户每周 N 小时算，这阶段几周）

3. **作品集清单**：列 2–4 件**能放进简历/能演示**的作品（按档位：S0 工具熟练度作品；S1 旧技能 ×AI 重做；S2 GitHub 可跑项目；S3 解决本行业真实痛点的 demo）。作品比证书值钱——它证明你真能干。

4. **公开发声与专业证据（推荐项）**：鼓励把学习和项目过程转换为可公开查看的产出。平台和形式不限：文章、图文、露脸视频、不露脸视频、纯声音、GitHub、Demo、Skill 讲解、功能教程、案例、工作流、AI 资讯、个人看法或使用感受都可以。计划应建议：
   - 主阵地和备份阵地；
   - 为什么该平台适合目标岗位；
   - 每个学习阶段要公开什么产出；
   - 可参考每周 1 次，但这是推荐而非硬性完成条件；
   - 每条内容对应哪项岗位能力或哪个作品证据；
   - 如何脱敏公司、客户和个人信息。
   不强制真名、露脸、粉丝数或提供公开链接。用户确认已发布即可记录，有链接时可自愿提供供分析。私人笔记和未发布草稿不算发声，但可算准备产出。

5. **求职时间线**：第几周开始投简历、投哪些渠道、面试要准备什么。明确"不必等全学完才投"——边投边补。

6. **止损线**：满足什么条件该停下复盘或调整（例：投 30 份 0 面试 / 学了 X 周完全跟不上 / 出现反诈红线）。给转型留个清醒的退出判断。

7. **诚实周期与预期**：以用户每周时间算，几个月到能投、几个月可能拿 offer。明说这是真实投入不是速成，不画饼。

> 需要时用 WebSearch 查该岗位当下的真实学习路径/主流工具栈/招聘要求（加年份），别给过时方案。

## 学习深度边界

计划的目标是达到当前岗位的最小可用能力，不是把每个主题学到专家级。对非核心技能，做到以下四点即可进入实践：

1. 能用一句话说清它是什么、解决什么问题；
2. 知道基本原理和输入—处理—输出关系；
3. 能在一个小案例中正确使用；
4. 知道怎样检查结果是否可用。

达到边界后停止纯理论深挖，转入项目、作品和投递。面试、项目或新 JD 暴露具体问题时，再做针对性补学。

Prompt 是所有 AI 转型者的通用基础能力，但同样服从上述边界：能说清原理、能结合工作使用、能验证和迭代即可，不得脱离目标岗位陷入无限学习。

## 工具选择原则

- 不得把 Coze 或 Dify 作为所有人第一天的固定任务。
- **Dify 是推荐的入门候选**：当用户需要直观理解 AI 应用的输入、模型、工具、知识库、工作流和输出时，可推荐 Dify，因为它可视化、低代码且公开学习资源较多。
- 只有当 Dify 与目标岗位、当前能力缺口或学习目标相关时才写入计划；不相关时不学。
- 其他工具同样以真实 JD 频次、岗位任务和用户当前差距为依据。

## 落盘（含「预期」——这是日后复盘对账的起点）
写入前先用系统时间取当前时间：
```bash
date '+%Y-%m-%d %H:%M %Z %z'
```
所有报告和状态记录必须带具体时间与时区，跟用户系统时区走。内部记录可用 `2026-06-26 22:06 CST +0800`；用户报告展示为 `2026-06-26 22:06（本机时区 + UTC 偏移，如 中国标准时间 CST，UTC+8）`。不要写死成某个固定国家的时间。

把策略写入 `.skill-state.json` 的 `active`（chosen_id / started_at=今天 / learning_plan），该岗位 status 改为 `learning`。
**同时写 `active.prediction`**：预计每周投入小时、预计几个月到能投简历、作品集清单、止损线（recorded_at=今天日期，recorded_at_full=具体时间和时区）。
这份预期写完别改——日后复盘拿"实际"和它对账，才能看清计划准不准、要不要调。

## 报告与存档收尾
学习计划生成后，不要直接结束。先问用户：

> "如果这版方向和计划没问题，我可以帮你存档并生成 Markdown 报告。如果你需要发送或打印，我也可以再导出 Word 版。你要现在生成吗？"

默认使用 Markdown 作为可迭代交付物，内容要用自然语言书写，不堆叠技术标记。Word 只在用户明确需要时生成。

若用户同意，生成两类内容：
- **内部归档**：继续写入 `.skill-state.json`；必要时生成 `reports/internal/<YYYY-MM-DD>-<target-slug>.md` 作为 agent 可维护的结构化存档，但不要把它作为主交付给用户。
- **用户阅读版**：生成 `reports/<YYYY-MM-DD>-<中文标题>.md`，用自然语言写成外行人也能读懂的完整报告。
  报告正文必须写 `生成时间：YYYY-MM-DD HH:mm（本机时区名 缩写，UTC 偏移）`，跟用户系统时区走，不要只写日期、也不要写死某国时间。示例：`生成时间：2026-06-26 22:06（中国标准时间 CST，UTC+8）`。

导出与交付规则：
- 默认生成 `reports/<YYYY-MM-DD>-<中文标题>.md`。
- 用户明确需要 Word 时，再将同一内容导出为 `.docx`。
- 告知实际生成的文件路径，不强制打开 GUI 文件夹。
- 报告中加入“公开发声建议”，但明确频率和链接提供都是非强制的。

报告交付后不要停在"文件已生成"。自然进入陪跑模式，但第一天任务必须来自已确认的学习计划。

> "从现在起，你可以跟着这份计划一起执行。我每天最多给你 3 件事，完成后会记入进度档案。如果你已有自己的规划，我会先核对它是否符合目标职业，再给建议。"

用户确认开始后，从 `learning_plan` 取出第一个里程碑，拆成当天最多 3 件事，并给出完成标准。没有已确认计划时，禁止套用任何固定工具教程。

报告结构建议：
1. 一页摘要：推荐方向、周期、目标薪资区间、首要作品。
2. 用户画像：起点档位、可迁移能力、约束条件。
3. 招聘信号：BOSS 搜索关键词、已看页数、已读 JD、薪资量级、岗位模式。
4. 推荐岗位簇：为什么选这个方向，哪些岗位作为锚点。
5. 差距分析：已有能力、要补能力、暂不硬冲能力。
6. 学习与投递计划：根据用户时间预算确定阶段、里程碑和每周任务，不写死固定周数。
7. 作品集规格：每个作品的输入、流程、输出、演示材料。
8. 求职动作：关键词、投递节奏、面试准备。
9. 止损线与反诈红线。
10. 下一次复盘问题。

## 普通收尾
- 提示用户：按阶段执行，到里程碑或卡住时回来调整（career-ai-transition-retro 待建，可先手动更新 retro_log）。
- 再次强调止损线和反诈红线：任何"交钱包就业/分期贷款学"的，立刻停。
