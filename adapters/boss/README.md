# BOSS直聘招聘侧 adapter（B 档：human-in-the-loop, read-only）

给 career-ai-transition 提供**真实招聘需求信号**（在招什么岗、要求哪些 AI 技能、同词岗位数量），补上通用网页检索拿不到一手平台数据、聚合报告又滞后的短板。

> ⚠️ **合规与边界声明（使用前必读）**
> 本 adapter 仅供**个人**在自己**已登录**的有界面浏览器里，对**公开搜索结果列表页**做**低频、只读**的辅助查阅，不替代你对 BOSS 直聘服务条款（ToS）与当地法律的遵守。
> - 只读你点名的页面：不自动登录、不自动翻页、不点"立即沟通"、不私聊、不批量抓取、不采集/保存 HR 等个人信息。
> - Chrome 调试端口对该独立浏览器实例拥有高权限；只在任务期间开启，完成后立即关闭 Chrome。
> - 不要在该独立 Chrome Profile 中登录邮箱、网银、云盘或其他敏感网站。
> - 调试端口只绑定 `127.0.0.1`，不得暴露到局域网或公网。
> - **不破解任何反爬机制**（含薪资字体混淆）——破解与本工具自身的反诈红线冲突。
> - 触发滑块/安全验证时由你**本人手动**完成，不做自动化绕过。
> - 请自行确认你的使用符合 BOSS 直聘 ToS；如平台规则变化或你不确定是否允许，请停止使用。本工具按 MIT 协议"按原样"提供，作者不对使用后果负责。

> 招聘数据可以作为当前市场需求信号。单次搜索是快照，不能单独证明岗位数正在上涨或下降。
> 对 career-ai-transition 来说，它不是单独下结论，而是给 career-ai-transition-scan 提供一手岗位信号，再和用户能力画像、AI 影响分类、可学性与反诈规则交叉判断。

## 它是什么 / 不是什么
- ✅ 是：读你**已登录、已搜索好**的 BOSS【搜索结果列表页】，把岗位卡片（`li.job-card-box`）提成结构化数据：岗位名 / 薪资 / 公司 / 地点（若列表页可见）/ 列表页标签（通常是经验/学历/实习周期等）/ 详情页链接。
- ❌ 不是：它**不自动登录、不自动翻页、不点"立即沟通"、不私聊、不批量抓详情**。导航、搜索、扫码、过滑块全是你（真人）在做；详情页只读用户明确选择的 1–3 个链接。
- **为什么默认只读列表页**：列表卡片只有**公开的岗位+薪资+公司+标签**，和闲鱼商品列表同一性质。详情页只在用户点名后读取 JD 文本，且不保存 HR 信息、不点沟通入口，保持 human-in-the-loop 边界。

## 机制（和闲鱼 adapter 的差异）
- 闲鱼 adapter 用 playwright 的 `connectOverCDP`。**本 adapter 不用 playwright**：它直接走**原始 CDP 协议**（`node:http` 取/建标签 + Node 22+ 自带的全局 `WebSocket` 发 `Runtime.evaluate`）。
- 原因：playwright 的 `connectOverCDP` 要做 browser-context 管理，跟新版 Chrome（实测 148）不兼容（报 `Browser context management is not supported`）。原始 CDP 只驱单个页面目标，绕开这个坑，**零依赖**。
- 副作用：`adapters/xianyu`（仍用 playwright）在 Chrome 148 上大概率也已经坏了，需要同样迁移到原始 CDP。

## 用法（3 步）
```bash
cd adapters/boss
# 无需 npm install —— 零依赖

./launch-chrome.sh               # 1) 起一个【有界面】Chrome(无头会被反爬挡!)
                                 #    默认使用 ~/.skill-chrome-profile
                                 #    若检测到旧 ~/.money-chrome-profile 会临时沿用旧登录态
                                 # 2) 在窗口里【扫码登录 BOSS直聘】(列表必须登录态才看得到)
                                 #    若弹滑块/安全验证,自己手动过一下
node read-boss.mjs "AIGC" 100010000 9222 1 # 3) 在登录态下搜该词、读第 1 页列表 → JSON
```
参数：`<关键词> [城市码,默认100010000=全国] [端口,默认9222] [页码,默认1]`。
输出的 JSON 直接喂给 `career-ai-transition-scan` 当招聘需求侧信号。

`launch-chrome.sh` 会把 Chrome 放到后台运行；关闭终端不会自动关浏览器。要结束会话，手动关闭这个 Chrome 窗口即可。
脚本会等待调试端口就绪；如果提示 `127.0.0.1:<端口> 暂不可读`，通常是已有普通 Chrome 被复用或端口冲突，先关闭旧 Chrome，或换端口运行 `./launch-chrome.sh 9223`。

## 只读用户选中的 JD
列表页只负责初筛。用户明确选择详情链接后，才读取 JD；可以选任意数量，也可以少选或不选。读取时保持低频顺序执行，不做无选择的批量抓取：

```bash
node read-boss-detail.mjs 9222 "https://www.zhipin.com/job_detail/xxx.html"
node read-boss-detail.mjs 9222 "https://www.zhipin.com/job_detail/a.html" "https://www.zhipin.com/job_detail/b.html"
```

边界：
- 只接受用户明确选择的 `https://www.zhipin.com/job_detail/...` 链接；一次很多时分批低频读取。
- 只提取岗位标题、公司、薪资、地点/地址、JD/要求文本、来源链接。
- 不读取/保存 HR 信息，不点"立即沟通"，不批量抓。
- 详情页失败或触发验证时，让用户手动在浏览器里过验证，或手动贴 JD。

## 先诊断再抓取
如果读不到 Chrome 调试端口，先跑：

```bash
node diagnose-cdp.mjs 9222
```

常见情况：
- `ok: true`：端口可读，可以继续跑 `read-boss.mjs`。
- `连接 127.0.0.1:9222 超时/拒绝`：Chrome 没用 `./launch-chrome.sh` 启动，或窗口已关。
- 在普通终端可读、在 Codex/Claude 沙箱里不可读：需要给读取本机端口的命令非沙箱权限。

## ⚠️ 薪资数字被 BOSS 字体混淆（重要，诚实标注）
BOSS 用自定义字体 `kanzhun-mix` 把薪资**数字**渲染成私有区码点（PUA, U+E000–U+F8FF）——浏览器里你能看见真数字，但 DOM 里抓不到。本 adapter **不破解这个反爬**（破解要下载字体做 OCR + 维护定期轮换的映射表，且本质是"规避平台审核"，与本工具反诈 rubric 的 A6 红线冲突）。

处理方式：**把混淆的数字位用 `▯` 占位，单位（K / 元/天 / 元/月）是明文照常保留**。例：
- `▯-▯▯K` → 个位~两位数 K 月薪
- `▯▯-▯▯K` → 两位数 K 月薪
- `▯▯▯-▯▯▯元/天` → 三位数日结（多为实习/兼职）

> 好消息：**位数结构泄露了量级**——足够区分"K 档月薪 vs 元/天日结实习"和大致档位，对需求信号判断通常够用。要精确数字 → 打开该条的详情页链接人工看。

## 已知限制
- **必须登录**：未登录列表页会跳转登录页（`diagnostics.maybeNeedLogin`）。
- **BOSS 频控比闲鱼凶**：搜太快弹滑块——因为挂的是你真浏览器，**你手动滑一下就过**。别改成高频批量（触发风控封号，且违反 ToS、与 A6 冲突）。
- **read-only、低频**：不自动翻页/沟通/投递。要更多结果时按页码少量读取，默认最多看 5 页。
- **BOSS 改版会让选择器腐烂**：`li.job-card-box` / `a.job-name` / `span.job-salary` 等若失效，输出会变空，`diagnostics.cardCount=0` 或落到 `sampleRawText`——把它发回即可重新校准。
- **每次读取会临时开一个独立标签页**：避免多个关键词同时跑时互相覆盖当前页面；读完会关闭该标签页。仍建议低频、少量关键词验证，不做批量采集。

## 怎么用这份数据（给 career-ai-transition-scan）
- **岗位名 + 列表页标签** → 先判断岗位方向、门槛、实习/全职属性；具体技能要求从用户选中的少量详情页读取。
- **同词岗位数量/趋势** → 涨=有人花钱、降=红海（趋势靠换时间或对比历史检索）。
- **薪资量级（▯ 位数 + 单位）** → 粗分 K 档月薪 vs 日结实习；精确值看详情页。
- **JD 标签** → 先判断这个岗位是在被 AI 增强、由 AI 新生，还是正在被 AI 替代。
> 仍要回到用户画像、AI 影响分类、可学性 rubric 和反诈 rubric 交叉验证，单一信号不下结论。
